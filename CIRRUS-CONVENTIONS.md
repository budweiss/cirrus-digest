# CIRRUS Conventions & Ground Truth

Purpose: this file is the single source of truth about how CIRRUS is actually
built. It exists because the qwen2.5:72b-generated proposals in
`digests/proposals/` were generic â€” they suggested frameworks and patterns
this project doesn't use. This doc gets fed into `generate_proposal()`'s
prompt so future proposals are grounded, and Claude (Cowork) should read it
before reviewing proposals or writing any code for CIRRUS.

## Hard architectural facts (do not contradict these in proposals)

- **Telegram bot (`cirrus_bot.py`) is a raw HTTP polling loop.** It calls the
  Telegram Bot API directly via `urllib.request` (`api_call()` â†’
  `getUpdates` / `sendMessage`), inside a `while True` loop with
  `time.sleep(5)` on error. It does NOT use `python-telegram-bot`,
  `Updater`, `Dispatcher`, or any bot framework. Proposals that rewrite the
  bot onto such a framework are not a fit â€” they'd be a full rewrite, not a
  scoped change.

- **All Ollama calls are plain HTTP `POST {OLLAMA_HOST}/api/generate`** with
  `{"model": MODEL, "prompt": ..., "stream": False}`, via `requests`. There
  is no `ollama` Python package import anywhere. "Fine-tune/retrain the
  Ollama model with feedback" is not something this setup does or easily
  could â€” don't propose model retraining.

- **Scheduling is via macOS `launchd`, not in-process loops.** Existing
  jobs: `com.cirrus.bot` (KeepAlive, runs the bot loop forever),
  `com.cirrus.daily` (7am daily digest), a weekly digest job (Sunday 7am),
  and `com.cirrus.api` (Flask API on port 5001). Anything that should run
  periodically gets its own plist in `~/Library/LaunchAgents/` and is
  triggered with
  `launchctl kickstart -k gui/$(id -u)/<job-label>`. Don't propose Python
  `schedule`-library loops or cron â€” they'd duplicate/conflict with launchd.

- **Config lives in `~/projects/cirrus-digest/config/sources.json`
  (`CONFIG["digest"]`, `CONFIG["email"]`, `CONFIG["podcasts"]`) and
  `config/credentials.json`** (secrets, gitignored). Both
  `cirrus_bot.py`, `cirrus_digest.py`, and `extract_actions.py` load these
  at import time via `Path.home() / "projects/cirrus-digest/config/..."`.

- **Directory layout** (all under `DIGEST_CFG["output_dir"]`, referred to as
  `digests/`):
  - `digests/daily-YYYY-MM-DD.md`, `digests/digest-*.md` (weekly)
  - `digests/actions/daily-actions-*.md`, `digests/actions/weekly-actions-*.md`
    â€” sections: `ACTION ITEMS`, `RECOMMENDATIONS`, `CIRRUS IMPROVEMENT NOTES`,
    `INTERESTING TOOLS/MODELS`, `FOLLOW-UP READING`
  - `digests/proposals/proposal-YYYY-MM-DD-N.md` â€” generated by
    `generate_proposal()` in `cirrus_bot.py`
  - `config/pending_approvals.json` â€” approval queue (`load_pending` /
    `save_pending` in `cirrus_bot.py`)
  - Logs in `LOG_DIR` (e.g. `bot.log`, `bot-error.log`)

- **Approval/extraction flow**: `extract_actions.py` parses a digest with
  Ollama into the actions file above. `cirrus_bot.py` re-parses
  `CIRRUS IMPROVEMENT NOTES` / `RECOMMENDATIONS` bullets into
  `pending_approvals.json` items (`type: CIRRUS_NOTE`, `detail`,
  `source_line`, `status`). Buddy reviews via `/approve` in Telegram
  (`approve N` / `reject N` / `approve all` / `reject all`). Approved
  `CIRRUS_NOTE` items go to `generate_proposal()`.

- **Deploy flow**: edit locally (in this Cowork folder, which mirrors
  `~/projects/cirrus-digest/` on CIRRUS) â†’ `scp` to CIRRUS â†’ `git add/commit/push`
  (repo `budweiss/cirrus-digest`, gitleaks pre-commit hook blocks secrets) â†’
  `launchctl kickstart -k gui/$(id -u)/<job-label>` to restart the affected
  service. No CI/CD beyond that.

## Style/scope conventions for proposals and code changes

- Prefer small, additive changes to existing files over rewrites. A "proposal"
  should be a diff-sized change to one or two functions, not a new framework.
- If a recommendation doesn't fit one of the patterns above, say so honestly
  ("no good fit yet") rather than inventing a plausible-sounding but
  incompatible design.
- Any new periodic behavior = a new or modified launchd plist + a script,
  not a long-running loop bolted onto an existing process (except the bot's
  own polling loop, which is intentional).
- Logging convention: `log(msg)` writes `[YYYY-MM-DD HH:MM:SS] msg` to stdout
  and/or `LOG_DIR / "bot.log"`. Follow this format for new log lines.
- Telegram replies go through `send_message(chat_id, text)`, which chunks at
  4000 chars and uses Markdown parse mode â€” reuse it rather than calling the
  API directly.
